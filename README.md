## 🧠 Опис функціоналу

Основна ідея проєкту полягає в тому, щоб спростити роботу з великими пакетами тендерної документації. У будівельних проєктах документи часто містять десятки або сотні сторінок PDF/DOCX-файлів, у яких важливі вимоги можуть бути розміщені в різних розділах: інструкціях для учасників, специфікаціях, додатках, формах, умовах договору або планах.

Система BidScop дозволяє:

- 🔐 реєструвати та авторизувати користувачів;
- 👥 створювати команди та керувати доступом учасників;
- 📁 створювати проєкти для окремих тендерів;
- 📄 завантажувати ZIP-архіви з PDF/DOCX-документами;
- 🧾 автоматично витягувати текст із тендерної документації;
- 🔎 знаходити важливі вимоги у документах;
- 🧠 класифікувати вимоги за допомогою ключових слів, регулярних виразів, семантичного пошуку та AI;
- 📌 зберігати знайдені результати в базі даних;
- 🟢🟠🔴 показувати статус знайдених вимог;
- 📅 відстежувати дати, дедлайни та інформацію про подачу тендеру;
- 📊 переглядати статус аналізу документів у реальному часі;
- 🛠️ керувати окремими функціями через панель адміністратора;
- ✉️ підтримувати функціонал, пов’язаний із надсиланням тендерних email-повідомлень.

## 🧩 Основна логіка аналізу документів

Аналіз документів виконується поетапно:

1. Користувач завантажує ZIP-архів із тендерними документами.
2. Backend приймає файл і створює задачу в черзі BullMQ.
3. Worker отримує задачу та починає обробку документів.
4. Документи класифікуються за типами: плани, специфікації, addendum, форми тощо.
5. Якщо потрібно, DOCX-файли конвертуються у PDF.
6. Для кожного документа запускається багаторівневий аналіз:
   - **1 рівень** — пошук за ключовими словами;
   - **2 рівень** — пошук за регулярними виразами;
   - **3 рівень** — семантичний пошук через embeddings;
   - **4 рівень** — AI-класифікація через Claude.
7. Знайдені вимоги зберігаються у таблиці `project_requirements`.
8. Frontend отримує оновлення через Supabase Realtime.

## ▶️ Як запустити проєкт "з нуля"

> Важливо: повний запуск можливий лише за наявності всіх частин коду та службових ключів. У публічному репозиторії розміщено тільки частину коду для тестування аналізу, тому деякі функції можуть бути недоступні без повної версії проєкту.

### 1. Встановлення інструментів

Перед запуском потрібно встановити:

- Node.js версії **22 або вище**
- npm версії **10 або вище**
- Git
- доступ до Supabase
- доступ до Upstash Redis
- API-ключ Anthropic Claude
- за потреби — Supabase CLI

---

### 2. Клонування репозиторію

```bash
git clone ...BidScop.git
cd BidScop
```

---

### 3. Встановлення залежностей

```bash
npm install
```

---

### 4. Створення `.env` файлів

Потрібно створити локальні файли середовища на основі прикладів:

```bash
cp apps/web/.env.local.example apps/web/.env.local
cp apps/api/.env.local.example apps/api/.env.local
cp apps/worker/.env.local.example apps/worker/.env.local
```

---

### 5. Приклад `.env` для frontend

```env
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=<your-supabase-anon-key>
VITE_API_URL=http://localhost:3001/v1
```

---

### 6. Приклад `.env` для API

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>

UPSTASH_REDIS_URL=rediss://default:<password>@<host>.upstash.io:6379

ALLOWED_ORIGINS=http://localhost:5173
PORT=3001

RESEND_API_KEY=<your-resend-api-key>
```

---

### 7. Приклад `.env` для worker

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>

UPSTASH_REDIS_URL=rediss://default:<password>@<host>.upstash.io:6379

ANTHROPIC_API_KEY=<your-anthropic-api-key>
AI_MONTHLY_BUDGET_USD=50
```

> ⚠️ Не можна публікувати реальні API-ключі, service role key, Redis URL або інші секретні дані у відкритому репозиторії.

---

### 8. Запуск усіх частин проєкту

```bash
npm run dev
```

Після запуску мають працювати:

- **Web**: `http://localhost:5173`
- **API**: `http://localhost:3001`
- **Worker**: підключається до Redis і очікує задачі аналізу

### 📤 Завантаження пакета документів для аналізу

**POST `/v1/analysis/upload-package`**

Призначення: завантаження ZIP-архіву з PDF/DOCX-документами для подальшого AI-аналізу.

**Логіка роботи:**

1. Користувач обирає ZIP-файл.
2. API приймає файл.
3. Створюється задача в черзі BullMQ.
4. Worker виконує аналіз документів.
5. Результати зберігаються в базі даних.

---

### 📊 Перевірка статусу аналізу

## 🚩 Feature Flags

У системі передбачено керування функціями через feature flags.

| Назва прапорця | Що контролює |
|---------------|--------------|
| `analysis.keywords` | Пошук за ключовими словами |
| `analysis.regex` | Пошук за регулярними виразами |
| `analysis.embedding` | Семантичний пошук через embeddings |
| `analysis.ai` | AI-класифікація через Claude |
| `analysis.upload` | Можливість завантаження документів |
| `project.delete` | Можливість видалення проєктів |
| `email.send` | Надсилання email-повідомлень |
| `team.registration` | Створення нових команд |
| `req.*` | Окремі типи вимог, наприклад BBD, PLA, STW тощо |

Пріоритет застосування:

```text
team override → global → enabled by default
```

## 🖱️ Інструкція для користувача

1. **Вхід у систему**
   - користувач відкриває вебзастосунок;
   - проходить авторизацію;
   - потрапляє на dashboard.

2. **Створення нового тендерного проєкту**
   - натиснути кнопку створення нового проєкту;
   - ввести назву проєкту;
   - додати інформацію про власника/замовника;
   - вказати дату подачі тендеру;
   - зберегти проєкт.

3. **Завантаження документів**
   - відкрити сторінку проєкту;
   - завантажити ZIP-архів із PDF/DOCX-документами;
   - запустити аналіз.

4. **Перегляд результатів аналізу**
   - після завершення аналізу система показує знайдені вимоги;
   - кожна вимога має статус:
     - 🔴 знайдено / підтверджено;
     - 🟠 результат невизначений або потребує перевірки;
     - 🟢 не знайдено.

5. **Робота з проєктом**
   - перегляд дедлайнів;
   - перевірка важливих вимог;
   - оновлення статусу проєкту;
   - прийняття рішення щодо участі у тендері.

6. **Адміністративні функції**
   - адміністратор може керувати feature flags;
   - переглядати статистику;
   - контролювати доступність окремих функцій системи.

## 🧪 Проблеми і рішення

| Проблема | Можлива причина | Рішення |
|---------|-----------------|---------|
| Проєкт не запускається | Не встановлені залежності | Виконати `npm install` |
| API не відповідає | Backend не запущений або неправильний порт | Перевірити `PORT=3001` і запустити `apps/api` |
| Frontend не бачить API | Неправильний `VITE_API_URL` | Перевірити `.env.local` у `apps/web` |
| Помилка Supabase | Невірний URL або ключі | Перевірити `SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` |
| Worker не запускається | Немає підключення до Redis | Перевірити `UPSTASH_REDIS_URL` |
| AI-аналіз не працює | Немає Anthropic API key | Перевірити `ANTHROPIC_API_KEY` |
| CORS помилка | Frontend origin не дозволений | Додати адресу frontend у `ALLOWED_ORIGINS` |
| Дані не оновлюються в реальному часі | Не налаштований Supabase Realtime | Перевірити налаштування Realtime для потрібних таблиць |
| Помилка через версію Node.js | Використовується стара версія Node | Встановити Node.js 22 або вище |
